《人月神话》读书笔记

发表于 2019-09-14 | 更新于 2019-09-18 | 分类于 读书笔记 |

焦油坑有个特点，当身陷其中时，挣扎的越猛烈，焦油就会缠的越紧。作者将大型系统开发比作焦油坑，旨在说明大型系统开发是一项极其繁杂困难的行为，不论团队规模大小，不论团队成员能力高低，都只有很少一部分可以从焦油坑走出来，即满足开发初期目标、进度、预算等要求。 这时人们迫切想寻找一把尚方宝剑，来解决开发管理中的各种问题，但很可惜，软件的特性决定了软件开发没有银弹，本书也只是从以下几个方面给出了软件工程管理上的一些建议。

团队构建

优秀程序员和较差程序员在编码效率上往往存在着很大的差异，所以管理人员倾向于使用小而精的队伍进行系统开发，但随之而来产生一个问题，即使效率再高的开发队伍，面对一个十分庞大的系统开发任务时，仍然会显得力不从心。

作者介绍了一种类似于“外科手术队伍”的解决方案，具体是指大型项目的每一个部分由一个团队负责，而每一个团队又以外科手术的形式构建，是一种 1+N 的形式。1 的那个人是一个经验丰富的系统架构师，负责整体的架构设计，而 N 进行工作上的支持。架构师还负责决定一切有争议的事务，所有的纷争都要由他来拍板。如果在这一点上不加以限制，团队的注意力会集中于各式各样的想法和争议上，也就无法专注于实现程序的具体功能。

编码人员可能会认为，一切都由架构师来决定，我们的创造力要如何体现？实际上程序的实现过程也能体现一个人的创造力，创造和发明的机会不会因为系统在主体上的限制而减少。此外，要保证团队中管理人员和技术人员具有相同的威信，管理人员要参与技术课程，技术人员要进行管理培训，保证这两类人在关键时刻可以互换。

软件开发团队之间要注重沟通和交流，如果交流不充分，人与人之间就无法产生有效的合作，甚至会产生矛盾与分歧，带来不小的麻烦。例如，有些小组会在工作过程中修改自己程序的功能，但并没有及时告知其它小组，最终导致整个项目出现混乱，开发无法正常进行。

进度控制

在众多软件项目中，缺乏合理的时间进度是造成项目滞后的最主要原因，而导致时间进度的规划出了问题，往往是由于对项目的规模没有一个清晰的认识。

有些人会用人月来衡量一项工作的规模，这暗示着人数和时间可以相互替换，即十个人做一个月的工作等同于一个人做十个月的工作，显然这是不可能成立的。因为软件开发是一个团队工作，人与人之间的交流沟通会耗费大量的时间，通过增加人手所减少的时间将不足以抵消因为交流沟通所增加的时间，最后的结果就是人多了，项目的进度反而更慢了。

在对项目规模的估计上还会存在这样的误区，例如利用比例估算整个项目的时间，或是用小型项目所需的时间去外推代码量更大的项目所需的时间。为了控制项目进度，可以设置“里程碑”，里程碑必须是具体且可度量的，模糊的里程碑为弄虚作假提供了可能，反而适得其反。

项目管理

有了上述项目开发过程中的“价值观”是远远不够的，还需要具体执行过程的“方法论”。

  • 手册是开发过程中很有必要的工具，手册中的描述必须是详尽且准确的
  • 会议也是项目运转过程中不可或缺的一部分，会议分为周例会和年度大会，周例会负责快速解决问题，而积压的老问题在年度大会中解决
  • 当实现人员遇到问题时，要积极与架构师进行沟通，而不是自行猜测，沟通的内容也要有书面记录，并发布给用户和其他实现人员
  • 产品测试很重要，测试要查明每一个可能的缺陷，因为所有的缺陷在现实使用环境中都会暴露出来

系统设计与开发

在系统设计过程中，概念完整性应该是最优先要考虑的因素，意思是指整个项目的设计思路应该保持统一，并且有时候为了统一，甚至要舍弃一些更高级的特性和改进。系统的概念完整性决定了系统的易用性。

架构师在设计第一个系统时，通常会比较谨慎，因为自己对整个项目的流程还不熟悉。但当他们准备设计第二个系统时，往往是信心满满，但这其中就埋伏着很大的隐患。他们想要将所有的好想法都塞进新系统中，但过于繁杂的功能反而会导致系统的易用性下降，有时候甚至还会影响到系统的基本功能。所以架构师在设计自己的第二个项目时，一定要小心谨慎，并时刻约束自己，不要对系统过度优化，画蛇添足。

另外，在项目设计初期就应该意识到，变化是不可避免的，无论是需求上的变化还是技术上的变化，架构师和编程人员都要做好应对变化的准备。例如对项目进行细致的模块化分割，设计函数时保证函数的可扩展性，撰写准确详尽的文档等。

编程人员在开发程序时，除了要注重程序的运行效率，也不能忽略了程序的运行空间，运行空间实际也是成本的一部分。项目经理在控制规模时，要考虑到所有的方面，而不只是程序的核心部分。当程序占用的空间成为一种问题时，要考虑是否通过削减功能或降低运行效率来保证程序的大小。

浅谈 Session 和 Cookie

发表于 2019-08-08 | 分类于 个人随想 |

为什么需要 Session 和 Cookie

HTTP 请求是无状态的，这也就意味着浏览器收到一个请求，只知道要做什么任务，却不知道是任务发布者是谁。举个例子，用户想要查看购物车信息，如果没有 Session，服务器就无法知道查看哪个用户的购物车，也就无法给出准确的答复。
Cookie 的数据保存在客户端，可通过浏览器进行查看；Session 的数据保存在服务端，相比较而言更加安全，但如果数据过多，会影响运行效率。

Session 和 Cookie 的合作

当浏览器发出一个请求时，Session 随之建立，同时返回给浏览器的 response 中包含Set-Cookie 字段，通常该字段中会存在 session_id（sid）值。浏览器收到 response 后，会根据 Set-Cookie 字段设置 Cookie，下一次发送请求时，会自动携带该 Cookie。
这样，下一次服务端收到请求后，根据 sid 找到对应 Session，从而也就找到了该会话的一系列状态信息，例如当前登录的用户等等。这也就解决了 HTTP 请求无法保存状态信息的问题。如果遇到浏览器禁止 Cookie 的情况，也可将 sid 附加到 url 中。
以下为本人猜测内容，未经考证。在成熟 Web 框架，例如 SSM 中，Controller 收到请求会根据 sid 自动解析 Session，因此可以通过 request.getSession() 函数直接获取 Session 信息，而不需关心与 Session 相关的底层操作。

如何实现“保持登录状态”这一功能

有些时候，我们需要实现这样一个功能，即退出浏览器下次访问该网站时，可以自动登录，不需要重新输入用户名和密码。这一功能的实现就依赖于 Session 和 Cookie。

  • 如果只靠 Session 来实现这一功能，可以延长 Session 的过期时间。一般来说，Session 会在会话结束时过期，即关闭浏览器后 Session 就会被销毁。而在 Django 中，Session 的默认过期时间是两周，如果用户不需要保存登录状态，则要将过期时间手动设为 0，这样关闭浏览器后用户就自动注销了。
  • 如果使用 Cookie 来实现这一功能，则无需延长 Session 的过期时间。当用户登录成功后，浏览器返回的 Set-Cookie 中不仅要包含基本信息，还要包含登录用户的 ID、Token 或相关信息。这样用户下一次访问该网站时，Cookie 信息会随 request 发送到服务端，服务端通过解析 Cookie 得到用户 ID 和 Token。再将 ID 和 Token 与存储在数据库中的信息进行比对，如果相符，则在 Session 中设置当前登录的用户信息，并跳转到主页面；如果不符，跳转到系统的登录页面。

《Core Java》读书笔记

发表于 2019-07-11 | 分类于 读书笔记 |

对象与类

Java 程序设计语言对对象采用的不是引用调用，实际上，对象引用是按值传递的。

一个方法不能修改一个基本数据类型的参数（即数值型或布尔型）。

一个方法可以改变一个对象参数的状态。

  • 方法得到的参数是对象引用的拷贝，即引用同一个对象

如果多个方法（比如， StringBuilder 构造器方法）有相同的名字、不同的参数，便产生了重载。

  • 举例：indexOf(int) indexOf(int, int) indexOf(String) indexOf(String, int)

继承

一个对象变量（例如，变量 e ) 可以指示多种实际类型的现象被称为多态（polymorphism)。 在运行时能够自动地选择调用哪个方法的现象称为动态绑定（dynamic binding )。

  • 当 e 引用 Employee 对象时，e.getSalary( ) 调用的是 Employee 类中的 getSalary 方法；当 e 引用 Manager对象时，e.getSalary( ) 调用的是 Manager 类中的 getSalary 方法
  • 一个 Employee 变量既可以引用一个 Employee 类对象，也可以引用一个 Employee 类的任何一个子类的对象
  • 使用多态方法或接口编写的代码比使用对多种类型进行检测的代码更加易于维护和扩展

不允许扩展（extend）的类被称为 final 类。如果在定义类的时候使用了 final 修饰符就表明这个类是 final 类。

类中的特定方法也可以被声明为 final。如果这样做，子类就不能覆盖这个方法。

  • final 的作用：阻止继承
  • 如果将一个类声明为 final，则其中的方法自动地成为 final，但不包括域
  • 若类中域为 final 域，则该域值不允许被修改

明确目的，不要被工具所困

发表于 2018-07-07 | 分类于 个人随想 |

「本文参与 少数派 2016 年度征文」

自从知道少数派网站以来，我对 iOS 应用就有了全新的认识。相比 Android 系统，iOS 的生态更加完善，app 种类更加丰富，而我也踏上了一条买买买的不归路。但在消费过后，我们不妨思考一下，有多少应用是我们真正需要的，它们的出现到底有无帮助？

效率工具真的提高效率了么？

效率工具，App Store 里的大热门，基本上每隔一段时间都会有新的效率工具出现。就拿 GTD 类应用来说，我手机上只有 Things 和 Clear 两个，而且我越来越发现，使用 Clear 的频率远远超过了 Things。以前，我很热衷于 GTD 类软件的探索，但后来我发现，它们对我效率的提高毫无益处。

Omnifocus， 大名鼎鼎的 GTD 应用，当然售价也是大名鼎鼎。少数派中有不少关于该 app 的文章，我留心了一下评论区，发现几乎在每篇文章下都有一些人是后悔购买的。为什么？我想是因为他们只看到了别人的使用经验，而忽略了自己的需求。拿我自己来说，我更多的只是需要一个清单类的应用，我只想知道我有哪些事需要做，在什么时候需要做，而不需要掺杂一些复杂的 GTD 理论，就我的需求来看使用 Omnifocus 这样的专业 GTD 工具成本未免太高了。

GTD，是「Getting Things Done」的缩写，最终目的还是要让事情 done，而我们可能花了更多的时间去寻找工具督促我们 done。我们将时间花在了无所谓的事情上，这样留给正经事的时间自然就少了，结果就是你对 GTD 理论有了很深的理解，但你仍然做不好一件事，拖延症仍无法摆脱。所以，明确目的很关键，找到满足需求的应用后，应该立刻投入到要做的事情当中，而不是在工具的海洋里迷失。
